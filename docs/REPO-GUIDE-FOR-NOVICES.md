# Your plain-English tour of the `brand-sites` repository

This document is for **readers who are not programmers**. By the end, you should be able to explain—out loud—what this repository is for, how the live websites get from here to the internet, and what the major folders do. A **complete list of every tracked file** appears in the appendix: each filename is a **hyperlink** to the matching file on GitHub so you can open the real source in one click.

> **Important scope note:** Git only stores the **human-written source** (the appendix counts **every path** returned by `git ls-files` at the time this guide was last regenerated). It does **not** store installed libraries (`node_modules`), build output (`.next`), or secret environment files (`.env.local`). Those exist on a developer’s laptop or on the hosting company’s servers after a build. When someone says “the repo,” they mean the tracked tree in the appendix.

---

## Part 1 — What this repository is

**GitHub** is a website that stores **versions** of a project. This project is named **`brand-sites`**. Think of it as a **shared filing cabinet** for everything needed to build and operate several **publication websites**—the marketing sites readers visit in a browser—plus related **editor tools**, **shared code libraries**, and a **copy of the subscription “magic” server** logic.

None of this *is* the live internet by itself. It is the **recipe and ingredients**. A hosting company (**Vercel**, in this setup) reads this repository, **builds** each website, and serves the result at real **domain names** you configure (for example a `.com` address). **Sanity** is a separate **content database** where articles and images usually live; the websites **fetch** that content when someone opens a page.

---

## Part 2 — What you “get” when this code runs

For each brand under `apps/`, the output is a **fast, modern marketing site** with:

- A **home page**, **about**, **archive**, **article pages**, **legal pages** (terms, privacy), **contact**, and **subscription-related pages** (thank-you, unsubscribe, snooze, profile, and similar flows tied to email links).
- A **header** with navigation and a **subscribe** experience (often a popup or embedded form with spam protection).
- **Ads** where configured, using shared components and sometimes brand-specific image assets.
- **Middleware** behavior: special links from email newsletters can land on the home page with query parameters; the site **redirects** those visitors to the correct internal page so URLs look clean.

The repo **does not** send newsletters by itself. Email delivery and subscriber databases are handled by other systems (**Customer.io**, **BigQuery**, and the **magic** serverless project). The marketing sites **call** those systems in a controlled way (for example posting a subscription form to an `/execute` endpoint).

---

## Part 3 — How a website is built (still plain English)

1. **Authoring:** Editors write articles in **Sanity Studio** (either a small app in this repo under `studio-*`, or a hosted studio). Content is stored in Sanity’s cloud.
2. **Configuration:** Each site has a `src/config/site.js` file—**one cheat sheet per brand** with display names, Sanity project identifiers, where the magic server lives, and similar constants.
3. **Page templates:** Under `src/app/`, each folder path maps to a **URL path**. A `page.js` file is the **template** for that address. Dynamic folders like `article/[slug]` mean “one template, many articles; slug is the piece of the URL that identifies the article.”
4. **Styling:** `.css` and `.module.css` files are **style sheets**—rules for colors, spacing, and fonts. “Module” means styles are **scoped** so they do not accidentally clash across pages.
5. **Shared logic:** Folders under `packages/` hold JavaScript that **more than one site imports**—for example talking to Sanity with the same query patterns, or executing subscription actions the same way everywhere.
6. **Build:** On a developer machine or on Vercel, a **build** step compiles everything into optimized files. That output is what visitors download.
7. **Deploy:** Pushing changes to GitHub triggers (or can trigger) Vercel to **rebuild** the affected site so the public domain shows the new version.

## Part 5 — Inside one `apps/<brand>/` folder (pattern repeats per site)

Each publication app follows the **same overall shape** so a developer can copy one site to start another:

- **`public/`** — Images and files served **as-is** at the site root (logos, favicons, phone mockups).
- **`src/app/`** — **Routes** (URLs) and global layout. Special files: `layout.js` wraps everything; `globals.css` is global styling; `middleware.js` (actually at `src/middleware.js`) handles redirects.
- **`src/components/`** — **Reusable pieces** of the user interface: header, subscribe popup, ad units, article chrome.
- **`src/config/site.js`** — **Brand constants** (names, ids, endpoints).
- **`src/context/`** — Remembers whether this browser is already subscribed (local storage backed).
- **`src/lib/`** — **Helper functions** (fetch articles, call subscription APIs, reader profile).
- **`src/data/`** — Small **static lists** or JSON used on the home page or promos.
- **`sanity/`** — Lightweight **schema definitions** used when this app’s Sanity CLI talks to the project.
- **Config files** — `next.config.mjs`, `eslint.config.mjs`, `jsconfig.json`, `package.json`: tooling and framework settings.

If you open the appendix and skim **one app** end-to-end, you will recognize the same **nouns** (pages, components, config) repeated with different **brand assets** and copy.

## Part 6 — Shared `packages/` in one breath

| Package folder | Role in one sentence |
|----------------|------------------------|
| `magic-client` | Talks to the magic server’s `/execute` endpoint and stores a short-lived **reader token** when the server returns one. |
| `platform-redirects` | Encodes the rules “if the home page has these query params, **redirect** to `/subscribed`, `/poll`, etc.” |
| `sanity-content` | Runs **GROQ** queries and maps Sanity documents into the article shape the pages expect. |
| `shared-ads` | Hosts **image files** and an `index.js` map so each brand can show consistent ad creatives. |
| `web-shell` | Small wrappers related to **third-party scripts** (for example ad or meta tags) used by sites. |

---

## Part 7 — `subscription-functions-copy/` (the “magic” side)

This tree is a **sibling world** to the marketing apps. It contains dozens of **API** files—each is a **mini program** that runs when a specific URL is hit on the deployed magic host. Together they:

- Accept **subscribe**, **unsubscribe**, and **snooze** actions from forms and email links.
- Talk to **BigQuery** tables of users and to **Customer.io** for email automation.
- Expose **`/api/reader-subscriptions`** so a logged-in-feeling profile page can list **cross-brand** subscriptions **without** exposing the marketing sites to raw database credentials.

The marketing sites are deliberately **thin clients**: they render UI and call **HTTPS endpoints**; they do **not** embed secret database keys in the browser.

---

## Part 8 — Studios (`studio-*`)

**Sanity** stores structured documents: articles, authors, images, etc. A **Studio** is the **editor interface** for that data. Each `studio-*` folder is a small TypeScript project defining **content types** (schemas). Editors run it locally or deploy a hosted studio; the **Next.js sites** only need project id + dataset in environment variables to **read** published content.

---

## Part 9 — How to sound like the author of this codebase (cheat sheet)

When explaining to someone else, you can say:

1. **“Monorepo”** — Many related projects in **one** Git repository so shared packages stay in sync.
2. **“Next.js app per brand”** — Each `apps/<name>` is an independent website codebase deployed to its **own** Vercel project (root directory set to that folder).
3. **“Sanity is the CMS”** — Articles and media live in Sanity; the site **fetches** them at build or request time depending on the route.
4. **“Magic is the secure subscription brain”** — Heavy lifting and secrets stay in `subscription-functions-copy` (deployed separately); browsers only see public keys and HTTPS calls.
5. **“Turbo + pnpm”** — Install once at the repo root; build or lint **all** packages in the right order, in parallel when safe.

That is the **architecture in spoken form**—the appendix below is the **inventory** that maps every tracked path to a short role.
